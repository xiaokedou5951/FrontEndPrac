# 架构文档组织方式

## 一、核心问题

一份架构文档变大的根因，往往是**把多个维度塞进一个文件**：

- 受众混在一起（产品 / 架构师 / 开发 / 运维各看一半）
- 稳定度混在一起（原则 vs. 决策 vs. 实现细节，更新频率差 10 倍）
- 抽象层级混在一起（系统全景 vs. 模块内部 vs. 代码级）
- "是什么 / 为什么 / 怎么做" 三件事都写在一篇里

拆分的目的不是"变多"，而是**让每篇文档只回答一个问题、只服务一类读者、只对应一种更新频率**。

---

## 二、四条拆分原则

| 原则 | 含义 | 判据 |
|---|---|---|
| 单一职责 | 每篇文档回答一个问题 | 标题能写成一句疑问句 |
| 受众分离 | 不同读者读不同文档 | 一段内容只有一个受众时，就该拆 |
| 稳定度分层 | 原则 > 决策 > 实现，分开存 | 内容更新频率差异 > 3 倍，拆 |
| 决策原子化 | 每个架构决策一份 ADR | 决策可被独立撤销 / 引用 |

**反例**：把"为什么选 PostgreSQL""表结构""索引策略""备份方案"写在一篇《数据架构》里 —— 四件事更新频率完全不同，迟早腐烂。

---

## 三、目录结构

```
docs/architecture/
├── README.md                  # 入口地图：导航 + 一句话架构 + 文档清单
├── 01-业务上下文.md           # 范围、利益相关者、核心场景
├── 02-架构总览.md             # C4 L1-L2：系统上下文 + 容器图
├── 03-模块设计/
│   ├── README.md              # 模块清单、边界、依赖矩阵
│   └── <模块名>.md            # 每个模块 / 服务一篇
├── 04-数据设计.md             # 存储模型、ER、关键数据流
├── 05-接口契约.md             # 对外 API、模块间契约
├── 06-横切关注点.md           # 安全、可观测、性能、容错
├── 07-演进路线.md             # 里程碑、技术债、风险
├── decisions/                 # ADR：每个决策一篇，只追加不修改
│   ├── 0001-<简述>.md
│   └── 0002-<简述>.md
└── glossary.md                # 术语表
```

**为什么这么分**：

- `01-02` 是**稳定层**，半年改一次，给所有人看
- `03-06` 是**设计层**，按模块 / 关注点切，给开发看
- `07` 是**变化层**，按版本演进
- `decisions/` 是**只追加层**，决策一旦记录永不修改，只能被新 ADR 取代

## 四、每篇文档的最小结构

```markdown
# <文档名>

> 一句话定位：本文档回答「<问题>」，受众是<谁>。

## 核心内容
（正文，控制在 300 行内）

## 相关文档
- [依赖的 ADR](./decisions/0001-xxx.md)
- [上下游模块](./03-模块设计/xxx.md)

## 变更记录
| 日期 | 变更 | 作者 |
```

**硬约束**：

- 每篇正文 ≤ 300 行；超出说明在混职责，继续拆
- 标题下必须有一句"定位"，写不出来说明文档目的不清
- 涉及"为什么这么做"的，链接到 ADR，不在正文展开

## 五、ADR（架构决策记录）模板

```markdown
# ADR-0001：<决策标题>

- 状态：提议 | 已接受 | 已废弃 | 已被 ADR-000X 取代
- 日期：YYYY-MM-DD
- 决策者：

## 背景
为什么要做这个决策？触发因素是什么？

## 决策
我们决定什么。一句话能说清。

## 备选方案
考虑过哪些方案？为什么没选？

## 后果
带来什么代价 / 限制 / 后续工作？
```

**规则**：ADR 只追加、不修改。推翻旧决策 = 新写一篇 ADR 标注"取代 ADR-00XX"。

---

## 六、何时拆 / 何时合

**拆**：

- 单文件 > 500 行，且无法再压缩
- 一篇文档里出现 ≥ 2 类受众
- 一篇文档里部分内容每周改、部分半年不动
- 一个模块的细节写进了系统总览

**合**：

- 为了凑目录把一个 30 行的小主题单独成篇
- 两篇文档互相重复 > 50%
- 一个 ADR 需要另一个 ADR 才能看懂 —— 直接在前者引用，不合并

---

## 七、维护规则

1. **README.md 是唯一入口**，任何人进来先读它，3 分钟内能找到自己要的文档
2. **改动必须更新变更记录**，无变更记录的改动视同没改
3. **ADR 永不删除**，只能被取代
4. **每季度体检一次**：检查死链、过时内容、该拆没拆的大文件
5. **文档 owner 制**：每篇文档在 README 清单里标注 owner，无人 owner 的文档 = 废弃

---

## 八、与 PRD / 代码的关系

```
PRD（业务需求）  ──回答"做什么"──►  产品文档
       │
       ▼
架构文档          ──回答"怎么设计"──►  本目录
       │
       ▼
代码 + 注释       ──回答"怎么实现"──►  仓库
```

架构文档**不写代码细节**（那是代码的职责），**不写需求细节**（那是 PRD 的职责），只写**设计决策与结构**。
