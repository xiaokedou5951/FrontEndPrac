# ADR-0003：展示组件与操作组件分离

- 状态：已接受
- 日期：2026-07-12
- 决策者：项目作者

## 背景

TokenBank 的 6 个业务组件中，有纯展示的（TokenBalanceCard、DepositBalanceCard）和含交易逻辑的（AllowanceCard、DepositCard、WithdrawCard）。

如果所有组件都自己调 `useWallet()` 并管理交易，会导致：
1. 展示组件不必要地成为客户端组件，增加客户端 JS 包体积
2. 交易逻辑分散在各组件内，测试和复用困难
3. 组件职责模糊——"显示余额"和"发起交易"是不同关注点

## 决策

将业务组件分为两类：

| 类型 | 特征 | 组件 |
|---|---|---|
| 展示组件 | 无 `'use client'`，纯函数式，所有数据通过 props 传入（含 `account`） | TokenBalanceCard, DepositBalanceCard |
| 操作组件 | `'use client'`，内部调 `useWallet()` 取 walletClient/publicClient，含交易逻辑 | AllowanceCard, DepositCard, WithdrawCard |

展示组件连 `account` 也从 props 接收（而非自调 `useWallet()`），确保完全不依赖客户端上下文。

## 备选方案

### 全部客户端组件

所有业务组件都加 `'use client'`，各自调 `useWallet()`。

**为什么不选**：
- 展示组件不必要地进入客户端 bundle
- `useWallet()` 调用点增多，Context 值变化时更多组件重渲染
- 违反"最小权限"原则——展示组件不需要 walletClient

### 全部展示组件 + 回调 props

操作也通过回调 props 传入：

```typescript
<AllowanceCard
  allowance={...}
  onApprove={(amount) => parentHandleApprove(amount)}
/>
```

**为什么不选**：
- 页面层需要承担所有交易逻辑，变得臃肿
- 交易状态（pending、txError）需要在页面层管理，然后传回组件，数据流变复杂
- 操作组件的 amount 输入状态也要提升到页面层，进一步膨胀

### 自定义 hook 封装交易逻辑

提取 `useApprove()`、`useDeposit()`、`useWithdraw()` hooks，组件调用 hooks。

**评价**：
- 这不是与当前方案互斥的，而是可以叠加的演进方向
- 当前 3 个操作组件的交易逻辑各不相同（校验规则、刷新范围），提取为通用 hook 的收益不大
- 当有 3+ 合约 Demo 时再提取 → [07-演进路线](../07-演进路线.md)

## 后果

**正面**：
- 展示组件可被 Server Component 直接渲染（如果未来需要 SSR）
- 操作组件自包含交易逻辑 + 本地状态（amount、pending、txError），页面层不关心
- 职责清晰：展示组件只管"怎么显示"，操作组件管"怎么交易 + 怎么显示"

**负面**：
- 操作组件不能被 Server Component 渲染（需要 `'use client'`）
- 操作组件的 props 接口比展示组件复杂（多了 `refresh: RefreshFns`）
- `account` 在展示组件中从 props 传入，在操作组件中从 `useWallet()` 获取，来源不一致（但这是刻意的——展示组件不依赖客户端上下文）

**设计约束**：
- 展示组件判断 `!account` → 显示"请先连接钱包"
- 操作组件判断 `!account || !walletClient` → canSubmit = false + handleXxx 直接 return
- 两类组件都处理了未连接状态，但处理方式不同（展示 = 占位文字，操作 = 按钮禁用）
