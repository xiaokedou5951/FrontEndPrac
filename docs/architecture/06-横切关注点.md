# 横切关注点

> 一句话定位：本文档回答「错误处理、轮询、配置校验、钱包重连等跨模块问题怎么解决」，受众是开发者。

## 错误处理

### 三层错误边界

| 层 | 错误类型 | 处理方式 | UI 表现 |
|---|---|---|---|
| 配置层 | env 地址缺失 | `configOk = false` | 页面显示配置缺失卡片 |
| 钱包层 | 无 provider / 用户拒绝 | `error` state | WalletBar 显示红色错误 |
| 交易层 | revert / 用户拒绝 | `txError` state | 操作卡片显示红色错误 |

### 配置错误

```
config/contracts.ts
  → env 为空或 isAddress 校验失败
  → tokenAddress / tokenBankAddress = null
  → configOk = false
  → configError = "缺少环境变量: NEXT_PUBLIC_TOKEN_ADDRESS, ..."

tokenbank/page.tsx
  → if (!configOk) return <配置缺失卡片 />
  → 不调用任何 hooks 的实际请求（enabled = false）
```

### 钱包连接错误

| 场景 | 处理 |
|---|---|
| `window.ethereum` 不存在 | `error = "未检测到钱包..."` |
| `requestAddresses` 用户拒绝 | catch → `error = e.message` |
| `getAddresses` 静默重连失败 | 清除 localStorage 标记 |

### 交易错误

操作组件统一模式：

```typescript
try {
  const hash = await walletClient.writeContract({...});
  const receipt = await publicClient.waitForTransactionReceipt({ hash });
  if (receipt.status === "reverted") throw new Error("交易执行失败（合约 revert）");
  // 成功 → 刷新
} catch (e) {
  setTxError(e instanceof Error ? e.message : String(e));
} finally {
  setPending(false);
}
```

> 不做 error toast 全局通知，错误就近显示在操作卡片内。

## 轮询策略

### 设计

| 数据 | 轮询间隔 | 原因 |
|---|---|---|
| Token 余额 | 4s | 交易后需及时更新 |
| 授权额度 | 4s | approve 后需更新 |
| 存款余额 | 4s | deposit/withdraw 后需更新 |
| Token 元数据 | 不轮询 | name/symbol/decimals 不可变 |

### 实现

每个 Hook 内部：

```typescript
useEffect(() => {
  if (!enabled || !account) { /* 清空 */ return; }
  setIsLoading(true);
  refetch().finally(() => setIsLoading(false));
  const interval = setInterval(refetch, 4000);
  return () => clearInterval(interval);
}, [enabled, account, refetch]);
```

### isLoading 语义

- `true`：仅首次加载（`data === null` 时）
- 轮询刷新不触发 `isLoading`（避免 UI 闪烁）
- 展示组件判断：`isLoading && data === null` → 显示"加载中..."

### 轮询的代价

- 3 个 Hook × 4s = 每分钟 ~45 次 RPC 调用
- 本地 anvil 可接受；生产环境应改用事件订阅 → [07-演进路线](./07-演进路线.md)

## 配置校验

### 校验时机

`config/contracts.ts` 在模块加载时执行（客户端首次 import）：

```typescript
const rawTokenAddress = process.env.NEXT_PUBLIC_TOKEN_ADDRESS;
export const tokenAddress: Address | null =
  rawTokenAddress && isAddress(rawTokenAddress) ? (rawTokenAddress as Address) : null;
```

### 校验内容

| 校验 | 失败结果 |
|---|---|
| 地址非空 | `null` → `configOk = false` |
| 地址格式合法（`isAddress`） | `null` → `configOk = false` |
| 两个地址都通过 | `configOk = true` |

### 下游消费

- 页面层：`if (!configOk)` → 显示配置缺失卡片
- Hook 层：`enabled = connected && configOk` → 不发请求
- 操作组件：`if (!tokenAddress || !tokenBankAddress) return` → 不执行交易

## 钱包重连

### 静默重连机制

```
首次挂载
  → localStorage.getItem("tokenbank:walletConnected") === "1"?
  → 是 → getEthereumProvider()
  → provider 存在 → getWalletClient(provider)
  → getAddresses()（不弹窗，仅查已授权账户）
  → addresses.length > 0 → getChainId() → 恢复状态 + attachListeners
  → 失败 → localStorage.removeItem(CONNECTED_KEY)
```

### 事件监听

| 事件 | 处理 |
|---|---|
| `accountsChanged([])` | 用户在 MetaMask 断开 → 清空状态 + 清 localStorage |
| `accountsChanged([addr])` | 切换账户 → setAccount(addr) |
| `chainChanged(hex)` | 切换链 → setChainId(parseInt(hex, 16)) |

> 不处理链切换的"正确性"校验（如用户切到非 31337 链），仅更新显示。生产环境应校验链 ID 并提示切换。

## writeContract 的 chain 参数

### 问题

`getWalletClient` 返回 `WalletClient`，其泛型为 `WalletClient<Transport, Chain | undefined, Account | undefined>`。当 chain 类型为 `Chain | undefined` 时，viem 的 `WriteContractParameters` 要求显式传 `chain`。

### 解决

所有操作组件从 `@/config/contracts` 导入 `chain`，在 `writeContract` 调用中显式传：

```typescript
import { chain } from "@/config/contracts";

await walletClient.writeContract({
  account, address, abi, functionName, args,
  chain,  // ← 显式传入
});
```

### 替代方案（未采用）

给 `getWalletClient` 加泛型 `WalletClient<Transport, Chain, Account>`，但这需要 `defineChain` 返回的具体类型而非 `Chain` 联合类型，增加复杂度。

## 未连接时的安全防护

| 层 | 防护 |
|---|---|
| Hook | `enabled = connected && configOk` → 不发 readContract |
| 展示组件 | `!account` → 显示"请先连接钱包" |
| 操作组件 | `!account || !walletClient` → canSubmit = false，handleApprove 直接 return |
