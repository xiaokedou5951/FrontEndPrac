# 横切关注点

> 一句话定位：本文档回答「错误处理、轮询、配置校验、钱包重连等跨模块问题怎么解决」，受众是开发者。

## 错误处理

### 三层错误边界

| 层 | 错误类型 | 处理方式 | UI 表现 |
|---|---|---|---|
| 配置层 | env 地址缺失 | `configOk = false` | 页面显示配置缺失卡片 |
| 钱包层 | 无 provider / 用户拒绝 | `error` state | WalletBar 显示红色错误 |
| 交易层 | revert / 用户拒绝 | `txError` state | 操作卡片显示红色错误 |
| 事件订阅层 | `getContractEvents` / `watchEvent` 失败 | `error` state | EventLogCard 显示错误 |

### 配置错误

```
config/shared.ts
  → env 为空或 isAddress 校验失败
  → tokenAddress = null

config/tokenbank.ts
  → tokenBankAddress = null
  → configOk = (tokenAddress !== null && tokenBankAddress !== null)
  → configError = "缺少环境变量: NEXT_PUBLIC_TOKEN_ADDRESS, ..."

config/nftmarket.ts
  → nftMarketAddress = null
  → configOk = (nftMarketAddress !== null)
  → configError = "缺少环境变量: NEXT_PUBLIC_NFT_MARKET_ADDRESS, ..."

各 page.tsx
  → if (!configOk) return <配置缺失卡片 />
  → 不调用任何 hooks 的实际请求（enabled = false）
```

### 钱包连接错误

| 场景 | 处理 |
|---|---|
| `window.ethereum` 不存在 | `error = "未检测到钱包..."` |
| `requestAddresses` 用户拒绝 | catch → `error = e.message` |
| `getAddresses` 静默重连失败 | 清除 localStorage 标记 |

### 交易错误

操作组件统一模式：

```typescript
try {
  const hash = await walletClient.writeContract({...});
  const receipt = await publicClient.waitForTransactionReceipt({ hash });
  if (receipt.status === "reverted") throw new Error("交易执行失败（合约 revert）");
  // 成功 → 刷新
} catch (e) {
  setTxError(e instanceof Error ? e.message : String(e));
} finally {
  setPending(false);
}
```

> 不做 error toast 全局通知，错误就近显示在操作卡片内。

### 事件订阅错误

`useNFTMarketEvents` 的 `init` 异步流程中，`getContractEvents` 或 `watchEvent` 可能因 RPC 不稳定、合约地址错、ABI 不匹配等原因失败。处理方式：

```typescript
try {
  const past = await publicClient.getContractEvents({...});
  // ... 处理历史事件
  unwatch = publicClient.watchEvent({...});
  setIsWatching(true);
} catch (e) {
  setError(e instanceof Error ? e.message : String(e));
  // isWatching 保持 false，EventLogCard 显示错误信息
}
```

> 订阅失败不影响页面其他功能（list/buy/cancel 仍可操作），仅 EventLogCard 显示错误。

## 轮询策略

### 设计

**TokenBank**：

| 数据 | 轮询间隔 | 原因 |
|---|---|---|
| Token 余额 | 4s | 交易后需及时更新 |
| 授权额度 | 4s | approve 后需更新 |
| 存款余额 | 4s | deposit/withdraw 后需更新 |
| Token 元数据 | 不轮询 | name/symbol/decimals 不可变 |

**NFTMarket**：

| 数据 | 轮询间隔 | 原因 |
|---|---|---|
| 上架列表 | 6s | list/buy/cancel 后手动 refetch 即时刷新，6s 兜底捕获他人操作 |
| 事件订阅 | 2s | watchEvent 轮询新区块，及时发现链上事件 |
| Token 元数据 | 不轮询 | 复用共享 hook，不可变 |

### 实现

每个轮询 Hook 内部：

```typescript
useEffect(() => {
  if (!enabled || !account) { /* 清空 */ return; }
  setIsLoading(true);
  refetch().finally(() => setIsLoading(false));
  const interval = setInterval(refetch, 4000);  // 或 6000
  return () => clearInterval(interval);
}, [enabled, account, refetch]);
```

### isLoading 语义

- `true`：仅首次加载（`data === null` 时）
- 轮询刷新不触发 `isLoading`（避免 UI 闪烁）
- 展示组件判断：`isLoading && data === null` → 显示"加载中..."

### 轮询的代价

- TokenBank：3 个 Hook × 4s = 每分钟 ~45 次 RPC 调用
- NFTMarket：1 个 listings 轮询 × 6s + 1 个 watchEvent × 2s = 每分钟 ~40 次 RPC 调用
- 本地 anvil 可接受；生产环境 TokenBank 也可改用事件订阅（NFTMarket 已采用）

## 事件订阅模式（NFTMarket）

### 双机制：历史 + 实时

```
useNFTMarketEvents 启动
  ├─ getContractEvents({ address, abi, fromBlock: 0n })
  │    → 一次性拉取所有历史事件
  │    → 解码 + 排序后追加到 state
  │
  └─ watchEvent({ address, events, pollingInterval: 2000, onLogs })
       → 持续轮询新区块
       → 命中事件就回调 onLogs
       → 解码后追加到 state 前部
```

### 去重

历史事件与 watchEvent 订阅可能重叠（订阅瞬间区块已出）。用 `useRef<Set<string>>` 按 `${txHash}:${logIndex}` 去重：

```typescript
const knownIdsRef = useRef<Set<string>>(new Set());
const addLogs = (incoming: MarketLog[]) => {
  const fresh = incoming.filter((l) => !knownIdsRef.current.has(l.id));
  fresh.forEach((l) => knownIdsRef.current.add(l.id));
  setLogs((prev) => [...fresh, ...prev].slice(0, MAX_LOGS));
};
```

### cleanup

组件卸载或依赖变化时：
- `unwatch()` 取消 watchEvent 订阅
- `cancelled = true` 阻止正在进行的异步回调写入 state

### viem API 约束

- `watchEvent` 只接受 `event`（单个 AbiEvent）或 `events`（AbiEvent 数组），**不接受完整 `abi`**
- `getContractEvents` 接受完整 `abi`，自动解码所有事件
- 故 `contracts/nftMarketAbi.ts` 同时导出 `nftMarketAbi`（完整）和 `nftMarketEvents`（仅事件数组）

### 独立监听脚本（HTTP + WebSocket）

前端 hook 用 HTTP 轮询 `watchEvent`（受 `pollingInterval` 限制）。另在 `viem-front/test/nftmarket/` 提供两个独立监听脚本，不依赖前端运行时：

| 脚本 | transport | 机制 | 延迟 |
|---|---|---|---|
| `watch-events.mjs` | `http()` | `eth_getLogs` 轮询 | 受 `POLLING_INTERVAL`（默认 1s）限制 |
| `watch-events-ws.mjs` | `webSocket()` | `eth_subscribe` 推送 | 毫秒级，事件入块即到 |

两者均：复用 `.env.local` 配置、回放历史事件 + 实时监听、颜色区分三种事件、`Ctrl+C` 打印汇总。WS 版额外内置心跳检测（仅状态变化时打印）和自动重连。

> 前端 hook 暂未迁移到 WS transport（viem `webSocket()` 在 hook 层的 reconnection 与 React 生命周期集成需额外处理），WS 版脚本已验证技术路径可行。

### 闭包类型收窄陷阱

`nftMarketAddress` 是 `Address | null`，在 guard 后 TypeScript 不会在 async 闭包中保留类型收窄：

```typescript
// 错误：闭包内 nftMarketAddress 仍被推断为 Address | null
if (!nftMarketAddress) return;
const init = async () => {
  await publicClient.getContractEvents({ address: nftMarketAddress }); // TS 报错
};

// 正确：捕获为局部 const
const marketAddress = nftMarketAddress;
const init = async () => {
  await publicClient.getContractEvents({ address: marketAddress }); // OK
};
```

## 配置校验

### 校验时机

三份配置文件在模块加载时执行（客户端首次 import）：

```typescript
// config/shared.ts
const rawTokenAddress = process.env.NEXT_PUBLIC_TOKEN_ADDRESS;
export const tokenAddress: Address | null =
  rawTokenAddress && isAddress(rawTokenAddress) ? (rawTokenAddress as Address) : null;

// config/tokenbank.ts
const rawTokenBankAddress = process.env.NEXT_PUBLIC_TOKENBANK_ADDRESS;
export const tokenBankAddress: Address | null = ...;
export const configOk = tokenAddress !== null && tokenBankAddress !== null;

// config/nftmarket.ts
const rawNftMarketAddress = process.env.NEXT_PUBLIC_NFT_MARKET_ADDRESS;
export const nftMarketAddress: Address | null = ...;
export const configOk = nftMarketAddress !== null;
```

### 校验内容

| 校验 | 失败结果 |
|---|---|
| 地址非空 | `null` → `configOk = false` |
| 地址格式合法（`isAddress`） | `null` → `configOk = false` |
| TokenBank：两个地址都通过 | `configOk = true` |
| NFTMarket：市场地址通过 | `configOk = true` |

### 下游消费

- 页面层：`if (!configOk)` → 显示配置缺失卡片
- Hook 层：`enabled = connected && configOk`（TokenBank）或 `enabled = configOk`（NFTMarket）→ 不发请求
- 操作组件：guard `if (!tokenAddress || !tokenBankAddress) return`（TokenBank）或 `if (!nftMarketAddress) return`（NFTMarket）→ 不执行交易

### `.env.local` 的生成

`.env.local` 可手填，但推荐用 [`viem-front/scripts/deploy-contracts.sh`](../../viem-front/scripts/deploy-contracts.sh) 自动生成：

- 按依赖顺序部署 4 个合约（MyERC20 → TokenBank / NFTMarket → SimpleNft）
- 用 `jq` 从 `contracts/broadcast/<Script>.s.sol/<chainId>/run-latest.json` 提取地址
- 用 `awk` 替换 `.env.local` 中的 4 行 `NEXT_PUBLIC_*_ADDRESS`（保留其他注释与 RPC/WS 配置；已存在则先备份为 `.env.local.bak.<时间戳>`）
- 部署完用 `cast call` 回读 `TokenBank.token()` / `NFTMarket.paymentToken()`，校验与 MyERC20 地址一致

这样可避免「合约已重新部署但 `.env.local` 未同步」或「TokenBank 构造的 token 地址与 `NEXT_PUBLIC_TOKEN_ADDRESS` 不一致」两类常见配置错误。详见 [部署脚本说明](../../viem-front/scripts/README.md)。

## 钱包重连

### 静默重连机制

```
首次挂载
  → localStorage.getItem("tokenbank:walletConnected") === "1"?
  → 是 → getEthereumProvider()
  → provider 存在 → getWalletClient(provider)
  → getAddresses()（不弹窗，仅查已授权账户）
  → addresses.length > 0 → getChainId() → 恢复状态 + attachListeners
  → 失败 → localStorage.removeItem(CONNECTED_KEY)
```

### 事件监听

| 事件 | 处理 |
|---|---|
| `accountsChanged([])` | 用户在 MetaMask 断开 → 清空状态 + 清 localStorage |
| `accountsChanged([addr])` | 切换账户 → setAccount(addr) |
| `chainChanged(hex)` | 切换链 → setChainId(parseInt(hex, 16)) |

> 不处理链切换的"正确性"校验（如用户切到非 31337 链），仅更新显示。生产环境应校验链 ID 并提示切换。

## writeContract 的 chain 参数

### 问题

`getWalletClient` 返回 `WalletClient`，其泛型为 `WalletClient<Transport, Chain | undefined, Account | undefined>`。当 chain 类型为 `Chain | undefined` 时，viem 的 `WriteContractParameters` 要求显式传 `chain`。

### 解决

所有操作组件从 `@/config/shared` 导入 `chain`，在 `writeContract` 调用中显式传：

```typescript
import { chain } from "@/config/shared";

await walletClient.writeContract({
  account, address, abi, functionName, args,
  chain,  // ← 显式传入
});
```

### 替代方案（未采用）

给 `getWalletClient` 加泛型 `WalletClient<Transport, Chain, Account>`，但这需要 `defineChain` 返回的具体类型而非 `Chain` 联合类型，增加复杂度。

## 未连接时的安全防护

| 层 | TokenBank 防护 | NFTMarket 防护 |
|---|---|---|
| Hook | `enabled = connected && configOk` → 不发 readContract | `enabled = configOk`（listings 不依赖 account） |
| 展示组件 | `!account` → 显示"请先连接钱包" | `!account` → 显示"请先连接钱包" |
| 操作组件 | `!account \|\| !walletClient` → canSubmit = false，handleApprove 直接 return | 同左 |
| 事件订阅 | — | 不依赖 account，连接前也订阅（便于观察链上活动） |
