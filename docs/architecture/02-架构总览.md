# 架构总览

> 一句话定位：本文档回答「系统由哪些部分组成、技术栈是什么、各部分如何连接」，受众是所有人。

## 技术栈

| 层 | 技术 | 版本 | 选型理由 |
|---|---|---|---|
| 框架 | Next.js (App Router) | 15 | 现代默认，SSR + 客户端组件 |
| UI 库 | React | 19 | Next.js 15 配套 |
| 类型 | TypeScript | 5.9 | strict 模式 |
| 样式 | Tailwind CSS | 4 | CSS-first 配置，PostCSS 插件 |
| 链交互 | Viem | 2.21+ | 直接使用，不引入 wagmi |
| 链 | Foundry (anvil) | — | 本地开发链，chain ID 31337 |

## C4 L1：系统上下文

```
┌─────────────┐       EIP-1193        ┌──────────────────┐       HTTP/RPC        ┌─────────────┐
│             │ ◄──────────────────► │                  │ ◄──────────────────► │             │
│  MetaMask   │   window.ethereum     │  viem-front      │   http://127.0.0.1    │   anvil     │
│  (浏览器钱包) │                       │  (Next.js app)   │       :8545           │  (本地链)    │
│             │                       │                  │                       │             │
└─────────────┘                       └──────────────────┘                       └──────┬──────┘
                                                                                       │
                                                                              ┌────────┴────────┐
                                                                              │                 │
                                                                              │  MyERC20        │
                                                                              │  TokenBank      │
                                                                              │  NFTMarket      │
                                                                              │  (已部署合约)    │
                                                                              │                 │
                                                                              └─────────────────┘
```

**关键交互**：
- 浏览器 ↔ MetaMask：通过 `window.ethereum`（EIP-1193）发起交易、请求账户
- 前端 ↔ anvil：通过 HTTP RPC 读取链上状态（余额、授权额度、存款余额、上架列表）+ 轮询订阅 NFTMarket 事件
- MetaMask ↔ anvil：MetaMask 连接到 localhost:8545 网络，签名后广播交易

## 分层职责

| 层 | 目录 | 职责 | 依赖方向 |
|---|---|---|---|
| 页面层 | `app/` | 组装组件、调用 hooks、传递 props | → 所有下层 |
| 钱包状态层 | `context/` | 管理连接状态、暴露 walletClient/publicClient | → lib, config/shared |
| 数据查询层 | `hooks/` | 读链上数据、轮询、手动刷新、事件订阅 | → context, config, contracts |
| 业务组件层 | `components/tokenbank/`、`components/nftmarket/` | 各功能专属 UI + 交易操作 | → context, config, contracts, ui |
| 共享组件层 | `components/shared/`、`components/ui/` | 跨功能复用的 UI（WalletBar、Button、Card 等） | 无（最底层） |
| 客户端工厂 | `lib/` | 创建 viem 客户端、格式化工具 | → config/shared |
| 配置层 | `config/shared.ts`、`config/tokenbank.ts`、`config/nftmarket.ts` | 读取 env、导出地址/链 | shared 无依赖；tokenbank/nftmarket 依赖 shared |
| ABI 层 | `contracts/erc20Abi.ts`、`contracts/tokenBankAbi.ts`、`contracts/nftMarketAbi.ts` | 手写合约 ABI 常量 | 无（最底层） |

## 关键架构特征

1. **客户端组件为主**：钱包交互依赖 `window.ethereum`，业务页面和操作组件均 `'use client'`
2. **按功能拆分资源**：ABIs、config、hooks、components 都按 `tokenbank/`、`nftmarket/`、`shared/` 三组组织，避免混合 → [ADR-0004](./decisions/0004-按功能拆分资源.md)
3. **页面层单次 hook 调用**：避免多卡片各自轮询同一数据 → [ADR-0002](./decisions/0002-页面层单次Hook调用与RefreshFns下发.md)
4. **展示与操作分离**：展示组件纯函数式，操作组件自取 walletClient → [ADR-0003](./decisions/0003-展示组件与操作组件分离.md)
5. **env 驱动地址**：不写死地址，重新部署改 env 即可；每个功能有独立的 `configOk` / `configError`
6. **事件订阅 + 轮询混合**：TokenBank 用轮询刷新余额；NFTMarket 用 `watchEvent` 订阅 + `getContractEvents` 拉取历史事件
